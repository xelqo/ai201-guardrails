# Provenance Guard

A backend system that any creative-sharing platform could plug into to classify whether submitted text was written by a human or generated by AI. It returns an attribution result, a calibrated confidence score, a plain-language transparency label, and supports a creator appeals workflow — with rate limiting and a structured audit log for production safety.

Built with Flask, Groq (`llama-3.3-70b-versatile`), pure-Python stylometry, Flask-Limiter, and SQLite.

## Running it

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
# create a .env with: GROQ_API_KEY=your_key_here
python app.py
```

The server runs on **port 5001** (port 5000 is occupied by macOS AirPlay Receiver on the dev machine). All example commands below use 5001.

Endpoints:
- `POST /submit` — classify a piece of text. Body: `{"text": ..., "creator_id": ...}`
- `POST /appeal` — contest a result. Body: `{"content_id": ..., "creator_reasoning": ...}`
- `GET /log` — read the audit log as JSON.

## Architecture overview

A submission flows through the system as follows:

```
POST /submit {text, creator_id}
        │
        ├─▶ Signal 1: LLM classifier (Groq)      → 0–1 score
        └─▶ Signal 2: stylometric heuristics      → 0–1 score
                        │
                        ▼
        combine:  ai_likelihood = 0.7·llm + 0.3·stylo
                        │
                        ▼
        threshold → attribution (likely_ai / uncertain / likely_human)
                        │
                        ▼
        build transparency label (plain-language text)
                        │
            ┌───────────┴────────────┐
            ▼                         ▼
   INSERT submissions row     INSERT audit_log row
   (mutable status)           (append-only history)
                        │
                        ▼
   response {content_id, attribution, confidence, label}
```

Appeal flow:

```
POST /appeal {content_id, creator_reasoning}
        │
        ▼
  look up submission by content_id  (404 if not found)
        │
        ▼
  UPDATE submissions SET status = 'under_review'
        │
        ▼
  INSERT audit_log row (original decision + appeal_reasoning)
        │
        ▼
  response {content_id, status: under_review, message}
```

**Storage.** Two SQLite tables with different jobs: `submissions` is the mutable current-state store (its `status` column is updated on appeal), and `audit_log` is append-only — every `/submit` and `/appeal` inserts a row and past rows are never modified, keeping the trail immutable.

## Detection signals

The pipeline uses two genuinely independent signals — one semantic, one structural — so they fail in different ways and the combination is more informative than either alone. Each outputs a 0–1 score where 1 = very AI-like.

### Signal 1 — LLM classifier (Groq, `llama-3.3-70b-versatile`)
- **Measures:** A holistic, semantic judgment of whether the text *reads* as human or AI-written (tone, coherence, overall feel). The model is prompted to return strict JSON (`{"ai_likelihood": <0–1>}`) at `temperature=0` for consistency.
- **Why chosen:** It captures stylistic and semantic coherence that simple statistics can't — the single most informative signal available for free.
- **Blind spot:** It is a black box and can be confidently wrong. AI-detecting-AI is itself an unreliable, unsolved problem, so it cannot be trusted alone.

### Signal 2 — Stylometric heuristics (pure Python)
Computes two measurable properties and averages them (weighted toward variance):
- **Sentence-length variance** (standard deviation of sentence word-counts). Humans mix very short and very long sentences; AI tends toward a steady medium length, so *low variance ≈ AI*.
- **Type-token ratio (TTR)** — unique words ÷ total words (vocabulary diversity).
- **Why chosen:** It measures the structural uniformity of text, a property that is independent of the LLM's semantic judgment and computable with no external libraries.
- **Blind spot:** It is unreliable on short text (too few sentences for meaningful variance; TTR is inflated on short inputs), and dense, uniform human writing can look statistically "AI-like." TTR in particular proved weak in testing (see Known limitations), so variance is weighted more heavily (0.7) than TTR (0.3) within the stylometry score.

## Confidence scoring

The two signal scores are combined with a weighted average:

```
ai_likelihood = 0.7 * llm_score + 0.3 * stylo_score
```

**Why 70/30:** The LLM judges meaning holistically and is the stronger signal; stylometry only flags statistical uniformity, which produces more false alarms (e.g. polished human writing). It is weighted lower as a supporting signal that nudges rather than decides.

**What the score means:** `ai_likelihood` is the probability the text is AI. Uncertainty lives in the middle of the range; confidence is implicit in distance from 0.5.

**Thresholds (deliberately asymmetric):**

| ai_likelihood | Attribution |
|---|---|
| `> 0.75` | likely AI |
| `0.4 – 0.75` | uncertain |
| `< 0.4` | likely human |

On a creative platform, labeling a human's work as AI is the worst outcome. The thresholds are intentionally asymmetric — it takes a high score (`> 0.75`) to flag AI but only a low score (`< 0.4`) to clear as human — so the system errs toward "uncertain" rather than a false AI accusation.

### Validating that the score is meaningful

I tested five deliberately chosen inputs spanning the range. The scores vary meaningfully (0.25 → 0.87) rather than clustering around a constant, and all three categories are reachable:

| Input | llm_score | stylo_score | confidence | attribution |
|---|---|---|---|---|
| Robotic corporate filler | 0.90 | 0.803 | **0.871** | likely_ai |
| Formal AI paragraph | 0.80 | 0.581 | 0.734 | uncertain |
| Formal human (economics) | 0.70 | 0.617 | 0.675 | uncertain |
| Lightly edited AI (remote work) | 0.42 | 0.636 | 0.485 | uncertain |
| Casual human (ramen review) | 0.20 | 0.378 | **0.253** | likely_human |

### Two example submissions

**High-confidence case (0.871 → likely AI):**
> "In conclusion, it is important to note that the implementation of best practices is essential. Moreover, organizations must leverage synergies. Additionally, stakeholders should optimize workflows. Furthermore, it is crucial to maximize efficiency across all domains."

Both signals agreed (LLM 0.90, stylometry 0.803 — uniform sentence length and connective-heavy phrasing), producing a high combined score.

**Lower-confidence case (0.253 → likely human):**
> "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after..."

Both signals agreed in the other direction (LLM 0.20, stylometry 0.378 — irregular sentence rhythm, casual diction), producing a low score.

Note that the two **borderline** cases (formal human at 0.675, lightly edited AI at 0.485) both land in "uncertain," which is the honest outcome: the signals disagree, and the system declines to make a confident call rather than risk a false accusation.

## Transparency label

The label shown to readers is plain-language and never exposes the raw confidence number, which would imply a false precision the detection does not have. The "likely AI" label is worded non-accusatorially because a real human may be behind it.

| Variant | Exact label text |
|---|---|
| **High-confidence AI** | "This content shows strong signs of being AI-generated. Our system is fairly confident, but automated detection isn't perfect — the creator can appeal this assessment." |
| **High-confidence human** | "This content appears to be human-written. No strong signs of AI generation were detected." |
| **Uncertain** | "We can't confidently determine whether this was written by a human or AI. Treat this result with caution." |

The label returned by `/submit` changes with the confidence score; all three variants are reachable (demonstrated in the scoring table above).

## Rate limiting

Applied to `POST /submit` with Flask-Limiter (in-memory storage, keyed per client IP via `get_remote_address`):

```
@limiter.limit("10 per minute;100 per day")
```

**Reasoning.** A real creator submits their own work occasionally — even a prolific writer rarely exceeds a handful of submissions per minute, so 10/minute never inconveniences legitimate use. The daily cap of 100 stops a script from cheaply flooding the (paid, Groq-backed) detection pipeline while remaining far above any human's realistic volume. The numbers are chosen to be generous to humans and hostile to automated abuse.

## Audit log

Every decision is written to the append-only `audit_log` table (structured SQLite, not console output). Each entry records: `timestamp`, `content_id`, `creator_id`, `attribution`, `confidence`, both individual signal scores (`llm_score`, `stylo_score`), `status`, and `appeal_reasoning`. Appeals add a second row for the same `content_id`, preserving the original decision while recording the contest.

Sample (a classification followed by an appeal of the same content):

```json
{
  "content_id": "95c6367c-2f02-4fbd-9aa0-81a132b75dd2",
  "creator_id": "t1",
  "attribution": "uncertain",
  "confidence": 0.71,
  "llm_score": 0.8,
  "stylo_score": 0.5,
  "status": "classified",
  "appeal_reasoning": null,
  "timestamp": "2026-06-29T23:42:02Z"
}
{
  "content_id": "95c6367c-2f02-4fbd-9aa0-81a132b75dd2",
  "creator_id": "t1",
  "attribution": "uncertain",
  "confidence": 0.71,
  "llm_score": null,
  "stylo_score": null,
  "status": "under_review",
  "appeal_reasoning": "I wrote this myself, non-native English speaker.",
  "timestamp": "2026-06-29T23:42:08Z"
}
```

The appeal row carries `null` signal scores because an appeal is not a re-classification — no new signals run; it only records the contest and flips status.

## Appeals workflow

Any appeal is accepted for a valid `content_id` (no identity verification). The creator provides `content_id` + `creator_reasoning`. On receipt the system looks up the original submission, updates its status to `under_review`, logs the appeal alongside the original decision, and returns a confirmation. Automated re-classification is not implemented — a human reviewer would later open a queue of `under_review` items showing the original text, the original verdict and confidence, and the creator's reasoning side by side.

## Known limitations

- **Very short submissions** (e.g. a haiku or a one-line caption). The stylometry signal breaks: sentence-length variance is meaningless with one sentence, and TTR is inflated on tiny inputs. The system falls back to a neutral 0.5 stylometry score (visible in the audit log for short test inputs), leaning entirely on the LLM — which is also unreliable on fragments. Such content should be treated as genuinely undetermined.
- **Dense, formal human writing** (e.g. academic or legal prose). Its uniform sentence structure makes the stylometry signal score it as AI-like — a false-positive risk. In testing, a formal human economics paragraph scored 0.675 (stylometry 0.617). The asymmetric threshold keeps it in "uncertain" rather than "likely AI," but it does not get correctly cleared as human. This is a direct consequence of stylometry measuring structure, not authorship.
- **TTR is a weak metric on short text generally**, which is why it is down-weighted relative to sentence-length variance within the stylometry signal.

If deploying for real, I would replace the hand-tuned stylometry normalization bands with thresholds calibrated on a labeled corpus, add a length-aware TTR correction, and require authentication on `/log`.

## Spec reflection

**One way the spec helped.** Writing the confidence thresholds and label text in `planning.md` *before* coding meant the `submit()` implementation was largely transcription — the threshold logic (`>0.75`, `<0.4`) and the three label strings came straight from the spec, with no mid-build guessing about what 0.6 should mean to a user.

**One way the implementation diverged.** The plan specified two stylometric metrics combined into one score, but did not fix the normalization details. In practice, the first attempt squashed all scores into a narrow band because TTR was pinned to 0 on short text (its diversity sat above the assumed ceiling). I diverged by retuning the normalization bands to the range real text occupies and weighting sentence-length variance more heavily (0.7) than TTR (0.3) inside the stylometry signal — a refinement the spec didn't anticipate but the data demanded.

## AI usage

I used an AI coding assistant as a pair-programmer throughout, always reviewing and testing output before trusting it.

1. **Stylometry signal + scoring logic.** I directed the AI to generate the second signal (sentence-length variance + TTR → 0–1 score) and the combination logic from my planning spec. It produced a working scaffold, but when I tested it the scores were squashed into 0.2–0.33. I had it print the sub-metrics, which revealed TTR was contributing nothing on short text. I then overrode the original equal averaging — retuning the normalization bands and re-weighting variance over TTR (0.7/0.3) — rather than accept the AI's reasonable-looking but flat output.

2. **Flask + endpoint scaffolding.** I directed the AI to generate the Flask app skeleton, the Groq signal function, and the `/appeal` endpoint. I revised several things: caught a typo it (and I) introduced in the stylometry return (`+ 0.3 +` instead of `+ 0.3 *`) that pushed scores above 1.0; reordered a startup `init_db()` call that was being invoked before its definition; and removed a duplicated route decorator. I also kept the 0.75 AI threshold despite obvious-but-formal AI landing in "uncertain," because that false negative is consistent with my false-positive-averse design rather than a bug to tune away.
